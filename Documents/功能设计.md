## 固件架构

```
┌─────────────────────────────────────┐
│  应用层 (Application Layer)          │
│  ├── 4路开关状态管理                  │
│  ├── 按键/触摸事件处理                │
│  ├── 继电器 + LED 联动控制            │
│  ├── 断电记忆功能 (NV 存储)           │
│  ├── 开关同步 (Z2M Binding)          │
│  └── 状态上报 (Reporting)            │
├─────────────────────────────────────┤
│  ZCL 层 (Zigbee Cluster Library)     │
│  ├── genOnOff Cluster × 4 (Endpoint)│
│  ├── genBasic Cluster               │
│  ├── genIdentify Cluster            │
│  └── genGroups Cluster (Binding)    │
├─────────────────────────────────────┤
│  Z-Stack 3.0 协议栈                  │
│  ├── 网络管理 (NWK)                 │
│  ├── 应用支持子层 (APS)              │
│  ├── 设备对象 (ZDO)                 │
│  └── 路由功能 (Router)              │
├─────────────────────────────────────┤
│  HAL 硬件抽象层                      │
│  ├── GPIO 驱动                      │
│  ├── 定时器                         │
│  ├── 中断管理                       │
│  └── NV 存储 (Flash)                │
└─────────────────────────────────────┘
```

## 本地控制

| 触发源 | 动作 | 结果 |
|--------|------|------|
| 触摸按键 1 | 按下 | 切换第1路继电器状态 + LED 反相 + ZCL上报 |
| 触摸按键 2 | 按下 | 切换第2路继电器状态 + LED 反相 + ZCL上报 |
| 触摸按键 3 | 按下 | 切换第3路继电器状态 + LED 反相 + ZCL上报 |
| 触摸按键 4 | 按下 | 切换第4路继电器状态 + LED 反相 + ZCL上报 |
| S1 短按 | — | 配网（HAL层处理） |
| S1 长按5秒 | — | 工厂复位 `bdb_resetLocalAction()` |

> **触摸信号**: WTC6106BSI 输出持续高低电平，触摸时 OUT 拉低，松开恢复高电平。100ms 轮询检测下降沿触发翻转，软件防抖（连续2次=200ms确认）。

## 远程控制 (Zigbee)

| 命令 | 动作 | 结果 |
|------|------|------|
| OnOff: On (EP1) | 第1路开 | 继电器吸合 + LED 灭 + ZCL上报 |
| OnOff: Off (EP1) | 第1路关 | 继电器断开 + LED 亮 + ZCL上报 |
| OnOff: Toggle (EP1) | 第1路翻转 | 翻转 + ZCL上报 |
| ... | ... | EP2-4 同理 |

## 状态上报

- 本地操作后主动上报状态变化（`zclSampleSw_ReportOnOffState()`）
- 远程命令处理后也上报
- 支持响应 Coordinator 的 Read 请求

| 步骤 | 操作 | 说明 |

## Router 路由功能

DIYRuZ_RT 固件默认以 **Router（路由器）** 角色加入 Zigbee 网络。本设备保留该功能，使 4 路智能开关同时具备 **Zigbee 信号中继** 能力。

| 优势 | 说明 |
|------|------|
| 扩展网络覆盖 | 开关安装位置通常分布在房间各处 |
| 增强网络稳定性 | 为终端设备提供多条路由路径 |
| 无需额外设备 | 不增加硬件成本 |
| 常供电优势 | 墙壁开关长期通电，不会因断电导致路由表丢失 |

主要配置参数：
```c
#define ZDO_COORDINATOR 0    // 不是协调器
#define RTR_NWK           1    // 启用路由功能
#define MAX_RTG_ENTRIES   40   // 路由表条目数
#define MAX_RREQ_ENTRIES  8    // 路由请求条目数
#define MAX_NEIGHBOR_ENTRIES 16 // 邻居表条目数
```

## 断电记忆功能（待实现）

### 功能说明

断电后再次上电时，自动恢复断电前的开关状态。

### 实现方案

CC2530 内部 Flash 可利用 NV (Non-Volatile) 存储机制保存开关状态。

**存储内容**:
```c
// NV 存储项: 4路开关状态 (1 byte, 每bit代表1路)
#define NV_DIYRuZRT_RELAY_STATE_ID  0x0402
uint8 RELAY_STATE;  // 4路状态打包在1字节中
```

**触发保存时机**: 每次开关状态变化后延迟 1~2 秒保存（避免频繁写入 Flash）

**上电恢复流程**:
```c
if (SUCCESS == osal_nv_item_init(NV_DIYRuZRT_RELAY_STATE_ID, 1, &RELAY_STATE)) {
    osal_nv_read(NV_DIYRuZRT_RELAY_STATE_ID, 0, 1, &RELAY_STATE);
}
```

| 注意点 | 说明 |
|--------|------|
| Flash 擦写寿命 | CC2530 Flash 约 10 万次擦写，延迟写入策略降低频率 |
| 写入延迟 | Flash 写入期间设备可能无响应，用定时器延迟写入避免阻塞 |
| 首次上电 | 无 NV 数据时默认全关 |

## 开关同步功能 (Binding)（待实现）

通过 Zigbee Binding 机制 + genGroups Cluster 实现。Z2M 端配置 Binding，设备端支持 genGroups Cluster 即可。

**设备端需要**:
1. genGroups Cluster (0x0004) — 已在 InClusterList 中
2. BDB_COMMISSIONING_MODE_FINDING_BINDING — 已启用
3. Binding 响应 — Z-Stack 3.0 自动处理

**工作原理**:
```
┌──────────┐   Binding Table   ┌──────────┐
│  开关 A   │◄────────────────►│  开关 B   │
│ EP1: OnOff│   直连通信        │ EP1: OnOff│
└──────────┘  (不经协调器)     └──────────┘
```

| 区域 | 页范围 | 大小 | 说明 |

| Boot Loader | 0~3 | 8KB | Serial Boot Loader |
| 用户代码 | 4~120 | ~234KB | 应用程序 + Z-Stack |
| NV 存储 | 121~126 | 12KB | 6 页 NV |
| Lock Bits | 127 | 2KB | Flash 锁定位 |

### OTA 分区规划

| 区域 | 页范围 | 大小 | 说明 |

| OTA Image | 61~120 | ~120KB | OTA 下载区（预留） |
| NV 存储 | 121~126 | 12KB | 不变 |

1. xcl 链接脚本注释标记 OTA Image 起始地址
2. OTA Cluster 注册: Endpoint 1 添加 `ZCL_CLUSTER_ID_OTA` (0x0019)
3. 编译开关: `#define ZCL_OTA 0`
